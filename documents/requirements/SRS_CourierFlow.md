# Software Requirements Specification

## CourierFlow — система доставки

## 1. Введение

### 1.1 Назначение
Документ определяет требования к backend API системы доставки CourierFlow в формате SRS.  
Документ предназначен для разработки, тестирования и приемки учебного проекта.

### 1.2 Область применения
CourierFlow поддерживает базовый процесс доставки:
- регистрация пользователей  
- выбор роли (`Customer` или `Courier`);
- создание заказов клиентами (`Customer`);
- просмотр и принятие заказов курьерами (`Courier`);
- обновление статуса доставки;
- отслеживание заказа по `tracking URL`.

### 1.3 Определения
- **API** — интерфейс взаимодействия клиентских приложений с сервером.
- **REST** — архитектурный стиль API на HTTP.
- **Customer** — пользователь, создающий заказ.
- **Courier** — пользователь, выполняющий доставку.
- **Order** — заказ на доставку.
- **Tracking URL** — уникальная ссылка для отслеживания состояния заказа.
- **JWT** — токен для аутентификации запросов.

## 2. Общее описание

### 2.1 Перспектива продукта
CourierFlow — backend-сервис, который может использоваться web/mobile-клиентом или иными инструментами (например, Postman).

Сервис хранит пользователей и заказы, предоставляет REST API для операций с заказами и отслеживания.

(Система учебная и намеренно упрощена)

### 2.2 Классы пользователей
1. **Гость (неавторизованный пользователь)**  
   Может зарегистрироваться/войти;

2. **Customer**  
   Создает заказы, просматривает свои заказы, отслеживает статус заказа.

3. **Courier**  
   Просматривает доступные заказы, принимает заказ, обновляет статус и текущее местоположение.

4. **Система**  
   Генерирует идентификаторы и tracking URL, хранит историю изменений заказа.

### 2.3 Ограничения системы
- Реализация только в формате REST API.
- Внутренняя авторизация (email/логин + пароль + JWT).
- Без платежных функций.
- Допускается хранение только последнего известного местоположения курьера для трекинга.

## 3. Функциональные требования

### 3.1 Регистрация и авторизация
- **FR1.** Система должна предоставлять endpoint регистрации нового пользователя с обязательным выбором роли: `Customer` или `Courier`.
- **FR2.** Система должна валидировать уникальность учетных данных (например, email).
- **FR3.** Система должна безопасно хранить пароль в виде хэша.
- **FR4.** Система должна предоставлять endpoint авторизации и выдавать токен доступа.
- **FR5.** Система должна определять роль пользователя из токена и ограничивать доступ к endpoint-ам по ролям.
- **FR6.** Система должна отклонять запросы без валидного токена к защищенным endpoint-ам.

### 3.2 Управление заказами
- **FR7.** `Customer` должен иметь возможность создать заказ на доставку.
- **FR8.** При создании заказа система должна присвоить уникальный идентификатор и `tracking URL`.
- **FR9.** Заказ должен содержать минимум: данные отправления/получения (в упрощенном формате), описание, текущий статус, владельца (`Customer`), назначенного курьера (если есть).
- **FR10.** `Customer` должен иметь возможность просматривать список своих заказов.
- **FR11.** `Customer` должен иметь возможность просматривать детальную информацию по конкретному своему заказу.
- **FR12.** Система должна хранить временные метки ключевых этапов обработки заказа.

### 3.3 Работа курьера
- **FR13.** `Courier` должен иметь возможность получить список доступных (непринятых) заказов.
- **FR14.** `Courier` должен иметь возможность принять один доступный заказ в работу.
- **FR15.** Система должна гарантировать, что один заказ одновременно может быть назначен только одному курьеру.
- **FR16.** После принятия заказа `Courier` должен иметь возможность изменять статус заказа по допустимому сценарию переходов:
  - `CREATED` -> `ACCEPTED` -> `PICKED_UP` -> `IN_TRANSIT` -> `DELIVERED`
  - допускается `CANCELLED` до завершения по правилам проекта.
- **FR17.** Система должна отклонять недопустимые переходы статусов (например, переход назад без предусмотренного правила).
- **FR18.** `Courier` должен иметь возможность обновлять текущее местоположение для активного заказа (упрощенное поле/координаты).

### 3.4 Отслеживание заказа (tracking URL)
- **FR19.** Система должна предоставлять публичный или ограниченно-доступный endpoint отслеживания по уникальному `tracking URL`.
- **FR20.** По `tracking URL` система должна возвращать текущий статус заказа.
- **FR21.** По `tracking URL` система должна возвращать последнее обновленное местоположение курьера (если доступно).
- **FR22.** По `tracking URL` система должна возвращать время последнего обновления статуса/местоположения.
- **FR23.** Система должна возвращать ошибку при обращении к несуществующему или некорректному `tracking URL`.

## 4. Нефункциональные требования (NFR)

### 4.1 Производительность
- **NFR1.** Среднее время ответа API для типовых запросов (без сложной аналитики) должно быть не более 500 мс при учебной нагрузке.
- **NFR2.** Система должна поддерживать одновременную работу не менее 50 активных пользователей в тестовой среде.
- **NFR3.** Операции чтения (список заказов, tracking) должны выполняться не медленнее операций записи при сопоставимой нагрузке.

### 4.2 Надежность
- **NFR4.** Система должна обеспечивать целостность данных при конкурентном принятии заказа курьерами.
- **NFR5.** При внутренних ошибках API должно возвращать стандартизированные коды ошибок и не терять уже сохраненные данные.
- **NFR6.** После перезапуска сервиса данные заказов и пользователей должны сохраняться в БД.
- **NFR7.** Ключевые действия (регистрация, создание заказа, изменение статуса) должны логироваться.

### 4.3 Безопасность
- **NFR8.** Все защищенные endpoint-ы должны требовать аутентификацию по токену.
- **NFR9.** Пароли пользователей должны храниться только в виде криптографического хеша.
- **NFR10.** Должны применяться базовые проверки входных данных (валидация обязательных полей, форматов, длины строк).
- **NFR11.** Пользователь должен иметь доступ только к разрешенным данным согласно роли.
- **NFR12.** Система не должна включать внешние механизмы авторизации (OAuth, соцсети, SSO) в рамках данного проекта.

### 4.4 Интерфейс (REST API)
- **NFR13.** API должно следовать принципам REST и использовать JSON для запросов/ответов.
- **NFR14.** API должно использовать стандартные HTTP-методы (`GET`, `POST`, `PATCH`) и корректные статус-коды (`200`, `201`, `400`, `401`, `403`, `404`, `409`, `500`).
- **NFR15.** Все endpoint-ы должны иметь единообразную структуру URL и формата ошибок.
- **NFR16.** Должна быть предоставлена актуальная спецификация API (например, OpenAPI/Swagger) для учебной приемки.

## Критерии приемки
- Реализованы роли `Customer` и `Courier` с разграничением доступа.
- `Customer` создает заказ и получает `tracking URL`.
- `Courier` видит доступные заказы, принимает заказ и меняет статус по допустимому сценарию.
- По `tracking URL` возвращаются актуальные статус и последнее местоположение.
- Все ограничения из раздела "Границы проекта" соблюдены.
